# DESIGN.md

## Product Name

Turtle Wishlist Tracker

## Product Summary

Turtle Wishlist Tracker is a lightweight project tracking system for the Turtle Research Team. It replaces a manual Word-based job tracking template with a structured web app.

The current manual process has several repeated areas:

* Work Log
* Jobs Kanban
* Job List
* Individual Job records
* Developer progress notes
* UAT / PROD status

The web app should make the task table the single source of truth and automatically generate the Kanban board, list view, dashboard summary, work log, and export reports from that data.

## Problem

The current project tracking template is useful but manual.

Main pain points:

1. Task progress needs to be updated in multiple places.
2. Kanban, job list, and job detail can become inconsistent.
3. There is no easy team login.
4. Team members cannot easily add or edit tasks themselves.
5. UAT and PROD status are difficult to track over time.
6. There is no structured export for reporting.
7. Comments and decisions are hard to preserve.

## Product Goals

The app should:

* Track Turtle Wishlist jobs in one place
* Allow team members to create and update tasks
* Show progress visually
* Reduce duplicate admin work
* Preserve decision history
* Support UAT and PROD release tracking
* Export data for reporting
* Be simple enough for non-technical users

## Non-Goals

This app should not become:

* A full Jira clone
* A sprint planning tool
* A complex agile management system
* A replacement for GitHub
* A replacement for the production turtle database

## Data Model

### profiles

Stores user role and display name.

### tasks

Main task table.

### task_comments

Stores discussion and updates.

### task_status_history

Stores status changes.

### task_attachments

Stores external links, screenshots, or file references.

## Task Code

Use readable task codes:

* T001
* T002
* T003

For MVP, task code can be manually entered or generated by a database function.

Preferred:

* Auto-generate next task code on creation.

## Status Logic

When a task status changes:

1. Update tasks.status
2. Update tasks.updated_at
3. Insert a row into task_status_history

Do not manually maintain a separate work log. The work log should be generated from status history and comments.

## Report Logic

Reports should be generated from database queries.

Examples:

### Weekly Summary

* Tasks created this week
* Tasks completed this week
* Tasks moved to UAT
* Tasks moved to PROD
* Open blockers

### UAT Summary

* All tasks with status = Under Test
* Include acceptance criteria
* Include owner
* Include related URL

### PROD Summary

* All tasks with status = Go Prod
* Include version
* Include commit URL
* Include PROD date

## Security Model

Use Supabase Auth and RLS.

Roles:

### admin

Can manage users and all tasks.

### developer

Can manage all tasks, comments, status, version, and commit fields.

### team

Can create tasks, edit their own tasks, and comment on tasks.

### viewer

Can only read.
